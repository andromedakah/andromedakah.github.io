The Gauge War

Wednesday, 15 July 2026 · Cross-sector edition · C-level + Engineering

Yesterday: govern the goal, not the gauge. Today the word "gauge" changes meaning. Per The Information — "Google and Microsoft Team Up to Beat Back Anthropic and OpenAI" — the enterprise incumbents made their move in the standards war for the agent-tooling layer. A bloc of Google, Microsoft, Salesforce, ServiceNow, Snowflake, Databricks, GitHub, Nvidia, Cisco and Hugging Face is rallying behind ARD (Agentic Resource Discovery) — the "phone book" that decides which tools an agent can even find — sitting right above Anthropic's MCP, the de facto connection default. Absent from ARD's backer list: Anthropic and OpenAI. It's the 19th-century railway gauge war with GPUs. Don't bet the estate on the winner — own the interchange.

1 · Executive Summary

The commoditization thesis found its clearest battleground: the standard that connects agents to your tools. This week the incumbents stopped fighting model-by-model and started fighting standards-by-standard. The vehicle is ARD (Agentic Resource Discovery) — a v0.9 open draft for how an agent discovers which tools and MCP servers exist before it connects. Publishers post ai-catalog.json at /.well-known/; registries index them; agents query at runtime. Backers: Google, Microsoft, Salesforce, ServiceNow, Snowflake, Databricks, GitHub, Nvidia, Cisco, GoDaddy, Hugging Face — and neither Anthropic nor OpenAI.

1. The battle moved up a layer — model to plumbing. Anthropic's MCP won the connection layer (de facto default in ~18 months). The incumbents contest the layer above — discovery — with ARD, and the layer beside it — agent-to-agent — with Google's A2A (~150 orgs, Linux Foundation). Whoever runs the registry decides which vendors an agent can surface. That's the switchyard, not the track.

2. This is lock-in risk, not a spec detail. GitHub already shipped agent finder — built on ARD, on all Copilot plans — so Copilot discovers and calls MCP servers and tools at runtime from catalogs you point it at. Convenient, and load-bearing: your agents' reach is defined by a registry someone else may control.

3. Adoption races ahead of the standard settling. Gartner: 40% of enterprise apps embed agents by end-2026 (from <5%); Salesforce Agentforce hit $800M ARR (+169% YoY) — even as 56% of CEOs report no significant AI benefit yet (PwC 2026) and Gartner expects >40% of agentic projects cancelled by end-2027.

Bottom line: in a gauge war the prize is never the best gauge — it's the junction everyone must pass through. Don't pick the winner; own the interchange (your registry, your allow-list, a neutral abstraction). Own the interchange, not the track.

2 · Allegory — The Gauge War

In the 19th century every railway laid its own track width. Brunel's broad gauge rode better — it didn't matter. A train built for one gauge couldn't run on another's rails, and at every break-of-gauge town, freight was unloaded and reloaded — a tax paid at every seam. The fight wasn't over the best gauge; it was over which one everyone would agree on, and the companies with the most track banded together to make theirs the standard, so every train terminated at their yards. The best design lost; the biggest coalition won.

MCP is the gauge Anthropic laid first; most track now uses it. ARD is the switchyard the incumbents just banded together to control — and the coalition is exactly the firms with the most track already in the ground. The two not invited to set the gauge are the two whose model business the coalition would like to route around. The barons who survived the gauge wars owned the interchange — the station that handled any train, whatever the standard.

The question it forces: If the standard our estate is built on lost the war in twelve months, how much track would we relay — and do we own an interchange (our registry, our allow-list, a neutral abstraction) that keeps our agents running whoever wins?

3 · Technical — The Layers of the War

The agent stack is a layered rail network. Connection (MCP, Anthropic): how an agent talks to a tool — the de facto default. Agent-to-agent (A2A, Google): how agents talk to each other — ~150 orgs, Linux Foundation. Discovery (ARD, the new front): how an agent finds which tools exist — ai-catalog.json, registries, runtime search. The play: MCP won connection for Anthropic, so the incumbents aren't attacking it — they're wrapping it. Own the discovery layer and you route the traffic, whoever laid the track.

5 · So What — 3 Moves for the Next 30 Days

1. Draw the gauge map. Inventory every production and shadow agent and record, for each, its connection standard (MCP?), discovery mechanism (a vendor registry? none?), agent-to-agent protocol (A2A?), and single point of failure if that gauge is deprecated. Same inventory the EU AI Act demands in 18 days — one artifact, two payoffs.

2. Own the interchange. Don't delegate discovery to a vendor's open catalog. Run an internal registry publishing an approved ai-catalog.json, and constrain runtime discovery so agents surface only what you permit — GitHub's own default. Dynamic discovery becomes a productivity win and an audit source.

3. Route through a neutral abstraction. Extend the 7 July router one layer up: a model- and standard-neutral layer between agents and the gauge, so switching connection or discovery standards is a config change, not a rebuild. Assume no winner; build the junction that doesn't care.
